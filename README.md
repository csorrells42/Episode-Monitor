# Episode Monitor

Standalone WPF camera-based low-motion episode monitor.

This app records observable sustained low-motion events, optional start/end snapshots, event video clips, symptom markers, and event-list CSV exports for review. It is a data-gathering tool only and does not diagnose a medical condition.

The right-side control panel is split into **Narcolepsy Tracking** and **Avatar Building**. Narcolepsy Tracking contains episode watch controls, thresholds, alert-baseline calibration, symptom capture, event-list export, the 30-day event-log viewer, and evidence output. Avatar Building contains the **This is Chris** subject check, avatar learning controls, **Open Avatar System**, tracking fidelity, DX12 preview, and camera controls.

The output folder is user-selectable and can be set to an external HDD. The app remembers the selected folder in `EpisodeMonitorSettings.json` beside the exe and in `%LOCALAPPDATA%\EpisodeMonitor\EpisodeMonitorSettings.json` so rebuilds do not silently fall back to local storage. On a fresh install with a D drive present, the first-run default is `D:\Episode Monitor Output`; otherwise it falls back to `EpisodeMonitorSessions` beside the exe. Detected events are saved into timestamped folders containing `event_summary.json`, `event_summary.csv`, landmark timeline files, and optional start/end snapshots or annotated `event_video.mp4` evidence depending on the selected checkboxes. The visible event list is stored in SQLite under `EventData\episode_events.sqlite` in the output folder; each database event row has one event folder and stores the snapshot/video paths when those files exist. Today's rows are reloaded when the app opens. **View Event Log** shows the last 30 days. The app prunes database rows and tied event folders older than 30 days on startup. Right-click an event in either grid to delete that one database row and its tied event folder/evidence.

For a guided avatar-collection launch from a command prompt in the repo, run:

```cmd
make-avatar.cmd
```

That script builds the app if needed, starts Episode Monitor with `--easy-avatar --output-folder "D:\Episode Monitor Output"`, opens the Avatar System dashboard, and requests easy avatar learning. It still does **not** auto-confirm the subject, turn the camera on, or bypass capture-quality gates; check **This is Chris**, turn on the camera, and let the overlay lock before measurements can be learned. The executable also accepts `--easy-avatar`, `--open-avatar-system`, `--start-avatar-learning`, and `--output-folder "D:\path"` directly.

The live preview includes an overlay showing the current tracking state, motion threshold, low-motion timer, and active event duration so the user can confirm what the monitor is treating as acceptable baseline versus an event cue. The same overlay is burned into saved event videos, and videos include up to 60 seconds of pre-event lead-in frames from the rolling buffer.

Face cue tracking adds local, offline eye-region and lower-face/jaw-region change scores compared with an alert baseline. Eye openness is the primary face cue; jaw/lower-face droop is treated as supporting evidence. The landmark aperture path also builds an alert baseline, then contributes eyelid-closure, mouth-opening, and structural jaw-contour droop cue scores to event triggers, overlays, event videos, and event summaries. A rolling landmark trend layer separately tracks whether average eyelid aperture is steadily falling and whether mouth/lip aperture is rising over the recent window, using robust median pairwise slopes so a brief noisy spike or backend handoff does not dominate the trend. This helps capture gradual onset instead of only final threshold crossings. Use **Calibrate Alert Baseline** only when awake, centered, alert, and not experiencing symptoms before relying on the eye/jaw cue thresholds. The saved alert baseline lives under `AlertBaseline\alert_baseline.json` in the output folder and is reused on the next app launch. The app does not nag for recalibration just because time passed; it asks for calibration when no usable saved baseline exists and only gently suggests recalibration if the camera/mode setup changes or tracking looks off after lighting, glasses, seating distance, or camera changes. These cues document visible changes only.

Event summaries include both the final-frame landmark state and event-level landmark aggregates such as minimum eye opening, maximum eyelid closure, maximum mouth opening/change, maximum mouth-opening velocity, maximum jaw-contour droop/change/velocity, maximum landmark cue score, strongest rolling eye-closing trend, strongest rolling mouth-opening trend, landmark sample count, sources, and backend statuses observed during the event. Each event folder also gets `landmark_timeline.json` and `landmark_timeline.csv`, a frame-by-frame evidence trail with motion, eye/mouth openings, jaw-contour droop, confidence/quality, reconstruction flags, artifact flags, cue scores, and trend slopes. Landmark metrics include raw aperture ratios, confidence-smoothed aperture ratios, structural jaw-contour droop normalized against face width, left/right eye asymmetry, eye agreement, possible one-eye artifact flags, eye/mouth measurement-quality percentages, cue eligibility flags, rolling trend slopes, and event-level quality aggregates so the trigger path is less sensitive to one noisy frame and easier to review afterward. Saved summaries and timelines also carry temporal face-lock reliability, continuity, eye reliability, mouth reliability, and capture-quality fields so a captured event can show whether the tracker was strongly locked, still warming up, limited, acceptable for measurement collection, or avatar-grade at the moment evidence was recorded. When MediaPipe is active, summaries and timelines also retain Face Landmarker blendshape corroboration such as left/right/average blink, jaw open, and mouth close percentages, plus baseline-relative blink/jaw/mouth-close changes when an awake MediaPipe baseline has been established. They also export MediaPipe eye/mouth opening correction ratios, corrected-frame booleans, corrected-sample counts, and maximum absolute correction magnitudes so reviewers can distinguish raw contour measurements from blendshape-stabilized working measurements. When the aperture backend supplies the eye or mouth contour, summaries also include image-quality diagnostics such as glare, contrast, sharpness, and dark-aperture coverage so glasses-heavy clips can be reviewed with context instead of treating every measurement as equally trustworthy.

The app also builds a local personal face model over time. This is not live neural-network retraining; it is a conservative, inspectable JSON profile of high-confidence personal baselines and ranges: face position/scale, head pose, eye opening, mouth opening, jaw-contour droop, MediaPipe blink/jaw-open/mouth-close blendshapes, aggregate eye/lip/jaw shape profiles, image-quality diagnostics, reconstruction/artifact counts, and measurement-only subject identity geometry. Live passive learning is gated by the **This is Chris - collect my face model** checkbox plus the **Start Avatar Learning** button, so the learning data does not silently collect from a spouse, child, guest, or anyone else using the computer. Saved models and measurement journals carry `SubjectId`, `SubjectDisplayName`, `SubjectCollectionMode`, and `UnknownSubjectPolicy`; the current policy is to reject unknown subjects rather than merge them into the same model. Before the personal model is allowed to learn, a capture-quality preflight must pass camera mode, face scale, eye/mouth evidence, temporal stability, glasses/artifact risk, subject-gate, and storage checks; low-resolution or low-frame-rate footage can still be reviewed, but it is not learned as long-term avatar material. The avatar system can still learn usual face range and motion during a narcolepsy event when the subject gate and quality gates pass; those samples are reduced-weight range/motion evidence, not alert-baseline calibration. Passive personalization is measurement-only by default: the app saves learned numbers, aggregate shape distributions, capture-quality verdicts, and rejects/accepts reasons, not continuous webcam images or video clips. Live runs save `PersonalFaceModel\personal_face_model.json`, `PersonalFaceModel\personal_face_motion_model.json`, `PersonalFaceModel\personal_face_corpus_readiness.json`, `PersonalFaceModel\personal_face_corpus_readiness.html`, `PersonalFaceModel\measurement_avatar_system.json`, `PersonalFaceModel\measurement_avatar_system.html`, `PersonalFaceModel\measurement_avatar_training_package.json`, `PersonalFaceModel\measurement_avatar_training_package.html`, `PersonalFaceModel\measurement_avatar_capture_plan.json`, `PersonalFaceModel\measurement_avatar_capture_plan.html`, and accepted measurement `.jsonl` files under the chosen output folder, so an external HDD can hold the long-running face-motion learning data with tiny storage needs. On startup or output-folder change, the app reloads `PersonalFaceModel\personal_face_model.json` from the selected folder and rehydrates the weighted distributions before accepting more samples, so the long-term model continues improving across restarts instead of starting over. Each saved model now includes a `LearningStability` block that reports accepted weight, anchor status, and maximum next-sample influence, making the slow weighted-update policy auditable for future avatar consumers. The live motion model is rebuilt from the newest accepted samples up to a bounded sample cap, filtered by subject id, and stores measurement-only velocity/coupling distributions for eyelid closing, mouth opening, jaw droop, head-pose movement, and MediaPipe blink/jaw/mouth blendshape motion so future animation code can use stable learned behavior without keeping passive webcam footage. Aggregate contour shape profiles are learned only from direct eye/lip/jaw observations, so reconstructed or artifact-suppressed contours do not become shape truth. User-facing pose readouts use XYZABC: X is horizontal frame position, Y is vertical frame position, Z is apparent camera-space depth, A rotates around X, B rotates around Y, and C rotates around Z. Dense landmark points and learned surface-profile points store X/Y/Z; A/B/C is stored on the frame, pose bucket, motion sample, or future local surface patch that interprets those points. The personal model stores compact pose buckets for front-neutral and A/B/C rotation coverage. The front-neutral bucket is preferred for neutral identity geometry, while turned-head buckets remain available for pose-aware correction and animation without being averaged into the same straight-on face shape. The learning-data health report scores whether the accepted measurements cover enough baselines, motion pairs, distance/face-size variation, head-pose range, explicit pose buckets, expression range, identity signature coverage, contour-shape coverage, direct eye-behind-glasses trust, direct mouth/jaw trust, measurement quality, capture-quality pass rate, avatar-grade capture rate, and storage headroom for extreme-accuracy avatar work, then suggests what to capture next. The avatar training package combines the neutral face profile, motion profile, identity profile, pose bucket profile, contour shape profiles, direct feature-trust scores, quality/capture-quality context, readiness scores, safety boundary, provenance policy, and source artifact names into one measurement-only contract for future 3D/avatar consumers. Early subject-gated previews can use a canonical face scaffold so the "digital me" view becomes inspectable quickly, but that scaffold is tagged as a low-trust template prior with its own contribution percentage. Template-prior geometry is never written back into the personal model, never counted as an accepted measurement, and never unlocks avatar training by itself.

Z remains an apparent camera-space value until zoom and physical distance are explicitly calibrated. When enough accepted subject-confirmed samples exist, the live overlay and Avatar System also show Z versus the learned reference face scale, such as `0.82x ref` or `1.35x ref`, derived from learned face width and eye spacing instead of guessed physical inches. New accepted samples also store compact Z evidence metadata: apparent Z units, learned-reference Z ratio when available, Z confidence, source kind, camera-FOV usage, learned-reference usage, and calibrated-distance status. The readiness report and batch audit show Z evidence health separately from broad face-size coverage so close/far motion cannot silently masquerade as a new face shape.

Storage policy for personalization:

Collection-audit output: live, evaluator, and synthetic stress runs also write `personal_face_collection_audit.json` and `personal_face_collection_audit.html` beside the other personal-model artifacts. This audit is measurement-only and explains attempted frames: face-lock rate, subject-gate state, event/calibration holds, personal-model accept/reject rate, capture-quality collectable rate, avatar-grade rate, top rejection reasons, top capture issues, and next actions. It stores no raw frames, no room imagery, no video clips, no full landmark meshes, and no face contours.

- Keep by default: weighted eye/mouth/jaw/pose/blendshape measurements, pose bucket summaries, aggregate eye/lip/jaw shape profiles, quality scores, reliability scores, aggregate distributions, and accept/reject reasons.
- Keep only during explicit event capture: event video, start/end snapshots, burned-in overlay video, event summaries, and landmark timelines.
- Keep only during explicit avatar-training sessions: high-quality source video or stills intended for future face/avatar training.
- Do not keep during passive learning: continuous webcam video, raw frame images, room imagery, per-frame contour dumps, or redundant per-frame mesh dumps unless a review/testing command explicitly asks for them.

Passive measurement journals are capped at `10,000,000,000` bytes by default. When the `PersonalFaceModel\measurements` folder exceeds that budget, the oldest daily `.jsonl` files are pruned first. The app enforces that budget when a measurement writes and when a saved output folder is loaded, so old oversized journals are cleaned before readiness reports or avatar packages use their storage totals. This keeps normal long-running learning in the KB-to-MB range for ordinary use, with a hard budget under 10 GB instead of filling a drive with 4K video. Raw video remains the high-value but storage-heavy artifact, so it should be captured deliberately with a clear purpose.

The avatar/3D-preview path builds on the same subject gate. The **Open Avatar System** button writes the latest face preview, learning-data health report, collection audit, avatar package, and capture plan, then opens one dark local dashboard at `PersonalFaceModel\measurement_avatar_system.html`. That page shows whether avatar learning is active, why it is or is not collecting measurements, how complete/useful the face learning data is, which pose buckets are covered, why frames are rejected, what the future avatar package would consume, and what to collect next. It also shows recent dense-mesh stability from the Last 10 Good Features cache. The current live guardrail is B-specific: once recent retained samples contain enough left/right head-turn range, avatar learning pauses if eyes, brows, nose, lips, or jaw slide in head-locked coordinates instead of rotating with the head. Camera distance toward/away from the camera, A-axis tilt, and C-axis tilt are intentionally tracked as separate problems so their corrections do not contaminate the B verdict; the Avatar System and Last 10 Good Features pages now report separate Z distance, A tilt, and C tilt lock health for inspection before those axes become stronger learning gates. This first preview is intentionally measurement-only and non-photoreal: it reconstructs a normalized neutral-face wireframe from learned identity ratios, eye opening, mouth opening, jaw-droop distributions, pose-normalized aggregate eye/lip/jaw contour shapes, and aggregate brow/nose/cheek/forehead surface profiles with Z evidence, then shows confidence and warnings. Camera distance and desk position are stored as capture/motion measurements, not used to shrink or stretch the preview head. Turned-head samples shape the feature outlines but are anchored back onto the neutral face instead of sliding the eyes, nose, mouth, or jaw sideways; when enough straight-on measurements exist, neutral face proportions come from the front-neutral pose bucket rather than from the full mixed-pose average. Identity locking uses a local owner profile only to answer "is this the enrolled subject?" Manual confirmation stays first, extreme warmup geometry mismatches are rejected once a usable signature exists, and stronger automatic rejection starts after the identity signature matures. Unknown or low-confidence faces must pause learning instead of creating a blended amalgamation of multiple people.

3D reconstruction is intentionally a sidecar contract, not WPF app logic. `Modules\Vision\Reconstruction` defines JSON work items and results for future workers such as the Deep3D/PyTorch reconstruction stack. That worker can live under WSL2 or Linux, consume only subject-gated measurement journals and explicit avatar-training media, then return meshes, coefficients, preview renders, and quality reports without making Episode Monitor depend on PyTorch, CUDA, or a Linux-only renderer.

Use **Capture Symptoms** when symptoms are active, recently active, or an episode feels like it is beginning before automatic thresholds fire. This marks the event list, locks alert-baseline calibration until one symptom-free hour has passed, starts sleepy-state event evidence recording when the camera is on, and saves the same database entry, overlay, summaries, and optional video/snapshots as automatic captures.

Camera mode discovery uses both Windows Media Foundation and bundled FFmpeg DirectShow probing. Camera enumeration follows the stronger Jericho Down camera catalog pattern: physical cameras exposed through both Media Foundation and DirectShow are merged into one picker row, with the DirectShow twin retained as the fallback/control path while Media Foundation remains the standard fallback capture path. Live preview can still use Windows Media Foundation directly so selected HD/4K modes are requested from the camera driver, then fall back to bundled FFmpeg if Windows capture cannot open the device. For selected non-Auto modes, the Media Foundation path must keep the requested resolution; if it silently falls back to a lower-resolution stream, Episode Monitor rejects that path and tries the bundled FFmpeg fallback instead. The mode picker prefers high-quality compressed camera modes such as MJPEG/H.264 when available, and the **Tracking fidelity** selector can choose a real HD or 4K input mode instead of leaving the driver on its low-resolution Auto default. The app now starts with a 4K camera target by default while limiting analysis frames to 1920px at 15 fps, preserving face/eye/mouth detail without making the visible preview carry every analysis cost. Media Foundation now emits raw NV12 camera frames immediately to the DX12 preview renderer and only builds the smaller WPF analysis bitmap when the tracking loop needs it, which reduces choppy fallback preview behavior. The default-on **DX12 preview viewport** attempts the Jericho Down native texture camera path first; if that path cannot open the camera, the retry cooldown is short and the standard Media Foundation/FFmpeg path can still use the DX12 BGRA/NV12 upload renderer.

Camera preview and data collection are intentionally separated. The camera path emits frames to the live preview as quickly as the camera and renderer can sustain; episode tracking, avatar learning, and landmark analysis consume the newest available frame asynchronously and may skip intermediate frames when processing is busy. The live landmark detector uses a latest-frame worker instead of a FIFO queue, so if dense face tracking falls behind during movement it drops stale analysis frames and catches up to the newest camera frame. Passive avatar measurements are compact JSON and are journaled on a slower cadence, so storage logging is not expected to limit live tracking speed. The live overlay is intentionally compact and throttled so WPF text layout does not become the next CPU bottleneck; full cue, audit, readiness, and quality details stay in the event summaries, timelines, CSV/JSON outputs, and Avatar System reports. The live wireframe preview can be head-locked to the eye-line and chin coordinate frame, which makes it easier to see whether features stay attached to the head during turns instead of sliding around in camera space. The avatar reports also separate contour/surface coverage from contour/surface Z profile health, so a nose, cheek, brow, eye, lip, or jaw line can be visible while still being marked as too flat or depth-unstable for high-confidence reconstruction; the preview renderer only turns learned Z into visible depth after confidence-gating those profile health scores.

The **Camera Controls** section uses the standard Windows DirectShow camera-control interfaces modeled after the Jericho Down camera stack. When the camera driver exposes them, it can show and adjust controls such as exposure, focus, zoom, brightness, contrast, sharpness, gain, and white balance. These controls include default ticks and a slight snap-to-default pull so lighting/focus/glare tuning can be repeatable without losing the dark UI.

The vision code uses a composite landmark tracker. The first backend is the local MediaPipe Face Landmarker Python sidecar under `Modules/Vision/MediaPipe/Sidecar`, using the model bundle under `dependencies/vision/dense-face-landmarks`. It is designed to output a dense 478-point face mesh that the C# mapper turns into eye, lip, jaw, and face contours. It also carries MediaPipe blendshape scores into the same evidence model, giving independent blink and jaw-open signals that can corroborate contour measurements when glasses, glare, facial hair, or shadows make eyelid/lip edges noisy. `Modules/Vision/Analysis/FaceLandmarkMetricCalculator.cs` keeps raw contour aperture values for audit, then uses MediaPipe blink/jaw-open/mouth-close evidence conservatively to cap false-open eyelid/lip contours or lift false-closed lip contours in the working smoothed metrics; the delta is exposed as correction evidence instead of being hidden. `Modules/Vision/Analysis/FaceLandmarkCueAnalyzer.cs` turns those same blendshapes into awake-baseline-relative blink and mouth evidence for cue scoring. If Python or the MediaPipe imports are not configured, the app reports the sidecar status and continues down the stack. The next model-backed slot is the bundled OpenCV LBF 68-point facemark backend under `dependencies/vision/opencv/facemark`; it maps model output into face, eye, lip, and jaw contours before the aperture fallback runs. Face localization can also use OpenCV YuNet under `dependencies/vision/opencv/yunet`; when `face_detection_yunet_2023mar.onnx` is bundled, YuNet gets first chance to find the face before Haar fallback. YuNet's eye points and mouth-corner points also seed or repair the fallback eye/mouth aperture boxes, which helps when glasses make Haar eye or smile detections unreliable. If multiple tiers lock on the same face, the composite layer fuses them: high-confidence MediaPipe/dense landmarks keep the face, eye, lip, and jaw geometry unless the model result is weak or artifact-suppressed, while the aperture fallback can still repair weaker model contours and lower-confidence LBF detections. If a model tier is missing or cannot lock, the app reports that backend status and continues down the stack. The fallback detects the face, refines eye and mouth aperture contours inside the detected regions, and emits the same landmark metrics that the model-backed tiers use. The state-of-the-art target remains a local MediaPipe/ONNX-style face-landmark model that outputs a dense face mesh, eye landmarks, mouth/jaw landmarks, facial asymmetry, expression blendshapes, head pose, and image-quality confidence into the same event pipeline.

The OpenCV LBF tier and aperture fallback both keep temporal face-lock continuity. After a good 68-point LBF lock, the app can carry a short decayed landmark hold through brief detector misses so the overlay and metric pipeline do not blink off immediately. The aperture fallback also tries local expanded re-detection before giving up, then provides a short low-confidence face hold. Before measuring, the fallback refines each predicted eye and mouth region by probing nearby boxes and choosing the strongest plausible aperture evidence, which helps when glasses, glare, or camera auto-follow make the first cue box slightly wrong. Aperture measurements now use a managed 3x3 smoothing pass plus a column-profile pass inside the detected eye or mouth opening, which averages many vertical samples across the feature instead of trusting a single bounding-box height; this reduces the influence of outlier pixels, glasses frame fragments, and glare artifacts while avoiding an observed native OpenCV blur crash in bad candidate regions. Landmark metrics measure eye and lip opening along each contour's own corner-to-corner axis, so C-axis head tilt does not inflate eyelid or mouth aperture the way a screen-axis bounding box can. For structural model contours such as LBF and MediaPipe dense mesh output, the metric averages paired upper/lower eyelid or lip gaps instead of letting one noisy landmark define the whole opening. Before landmark metrics are calculated, a temporal reconstruction stage repairs likely glasses/occlusion artifacts. It can infer a missing eye contour from the paired eye, mirror first-frame paired-eye geometry around the face center when no prior frame exists, suppress abrupt one-eye aperture spikes, suppress shifted or oversized glasses/glare contours even when their aperture ratio looks plausible, remember the last safe eye geometry instead of the rejected artifact geometry, prevent low-fidelity fallback contours from rapidly reopening eyelids without stronger evidence, and carry forward short low-confidence mouth aperture gaps. Reconstructed left-eye, right-eye, mouth, and eye-artifact-suppression flags are carried into metrics, overlays, event summaries, CSV output, and offline evaluator summaries so review files can distinguish directly observed contours from inferred continuity. In live and burned-in event overlays, directly observed contours draw solid, reconstructed contours draw dashed, and artifact-suppressed eye contours draw dashed amber. This is not a substitute for real footage validation, but it makes the proof-of-concept better at tracking gradually closing eyelids and jaw/lip opening across imperfect webcam frames.


That stress run feeds moving, scaled, C-tilted synthetic landmark faces through the same temporal reconstruction, metric calculator, cue analyzer, trend analyzer, event aggregate, and landmark timeline code used by the app. It intentionally injects missing-eye frames, shifted glasses/glare false contours, and short mouth-shadow gaps. The verifier fails if eyelid opening does not slope downward, mouth opening does not slope upward, structural jaw droop does not slope upward, reconstruction/artifact-suppression evidence is not exercised, or aggregate/timeline evidence loses frames.

The stress run also writes `personal_model\personal_face_model.json`, `personal_model\personal_face_motion_model.json`, `personal_model\personal_face_corpus_readiness.json`, `personal_model\personal_face_corpus_readiness.html`, `personal_model\measurement_face_preview.json`, `personal_model\measurement_face_preview.html`, `personal_model\measurement_avatar_training_package.json`, `personal_model\measurement_avatar_training_package.html`, `personal_model\measurement_avatar_capture_plan.json`, and `personal_model\measurement_avatar_capture_plan.html`, then verifies that personalization accepts stable high-quality frames while rejecting event-like frames as symptoms progress. This is the foundation for "improves over time": the current implementation adapts transparent per-person baselines, ranges, readiness guidance, motion distributions, a measurement-only avatar package, and a prioritized capture plan rather than silently changing neural model weights.

Digital legacy/avatar boundary:

Episode Monitor's face-motion learning data may be useful to a future local legacy avatar or "digital self" project, but that product must keep a clear identity rule: whenever it identifies itself, starts a session with a new person, communicates outside a trusted private context, or touches legal, medical, financial, identity, authorization, or agency topics, it must say it is a digital representation of a real person and not the living person. In private companion mode that disclosure can be handled through onboarding, settings/about text, and contextual reminders instead of repeatedly interrupting grief-sensitive conversation, but the system must never hide what it is, claim authority as the real person, call institutions as the person, or invent certainty.

For unattended testing, use the soak runner:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\RunEpisodeMonitorVisionSoak.ps1 -Iterations 12
```

The soak runner repeats the full verifier, captures one `verify.log` per iteration, isolates optional sample-media output per iteration, and writes `vision_soak_summary.json` under `output\vision-soak\<timestamp>`. Each iteration summary includes sample, synthetic, and landmark-stress face reliability, continuity, eye usability, mouth usability, and overall-quality fields so unattended runs show whether tracker quality is drifting even when every iteration passes. Add `-StopOnFailure` if you want the run to stop at the first failure. This is useful for catching intermittent native crashes, sidecar startup problems, or unstable face/eye/mouth/jaw evidence while the machine is unattended.

For several real webcam clips, use the batch evaluator:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\RunEpisodeMonitorRealClipBatch.ps1 -SampleMediaPath "D:\clip1.mp4","D:\clip2.mp4","D:\clip3.mp4"
```

The batch evaluator keeps each clip's normal `vision_eval_report.html`, `vision_eval.csv`, overlay frames, summary, personal learning-data health JSON/HTML, avatar package, and capture plan, then writes one `real_clip_batch_report.html` and `real_clip_batch_summary.json` under `output\vision-real-clips\<timestamp>`. Use this after real testing sessions to compare face-lock range, temporal face reliability, warmed lock continuity, eye/mouth usable rates, quality, capture-quality score, avatar-grade frame rate, MediaPipe dense-lock rate, reconstruction counts, one-eye artifact flags, learning-data readiness percent, identity-coverage percent, contour-shape coverage percent, direct feature trust percent, subject-mismatch rejects, motion-pair count, next-capture suggestions, first recommended capture-plan item, capture-plan target minutes/bytes, and the weakest overlay frames across multiple distances, head positions, glasses glare, and camera auto-follow movement. The combined HTML report now selects the lowest warmed face-reliability frame and lowest capture-quality frame for each clip, so the review focuses on meaningful post-warmup lock weakness and conditions that are not good enough for long-term avatar learning. It defaults to no eye inset for normal webcam clips; pass `-EyeInset auto` or `-EyeInset bottom-right` for a clip that includes a zoomed eye reference.

Add `-FailOnQualityGates` when you want the batch to exit non-zero unless the real clips clear the machine-readable proof gates: face lock, eye/mouth usable rates, face reliability, capture quality, avatar-grade evidence, capture-plan output, same-subject protection, measurement-storage estimate under 10 GB, and optional eye-inset measurement/diagnostic coverage. For multi-clip batches, the strict gates also require meaningful horizontal face movement, vertical face movement, and face-scale/distance variation so the proof covers a face that moves around the frame instead of only a centered still subject. The HTML review frame section includes overlays for leftmost/rightmost, highest/lowest, closest/farthest Z apparent scale, and lowest/highest A/B/C rotations so the edge cases behind those gates can be inspected visually without translating older pose vocabulary.
